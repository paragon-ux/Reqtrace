@reqtrace annotation in source code

The core marker: @reqtrace REQUIREMENT/ORDINAL/@file placed in comments near evidence
Keep ▾

Necessary?

Yes. The annotation is the entire foundation. Without it there is nothing to grep, nothing to validate, and no evidence chain. Everything else is scaffolding around this marker.

Friction?

Low in practice. Adding a comment line near evidence is the same effort as any other code comment. The syntax is verbose but it is placed once and never touched again unless the requirement changes.

Value vs. steps?

Very high ratio. One comment makes a code location findable by requirement handle forever, without any tooling, in any editor, by any developer or agent.

The annotation is the strongest part of Reqtrace. The design decision to put the marker in a comment rather than structured metadata (YAML frontmatter, sidecar file, external DB) is correct — it travels with the code, survives merges and moves better than any external record, and is immediately readable by both humans and grep. The choice to use @ as the prefix is sound: it avoids collisions with language keywords, is recognizable as metadata, and works across every comment syntax.

The one tension is that the annotation is in two forms simultaneously: the unresolved form in code (@file) and the resolved form in the ledger (actual path). This split creates a synchronization requirement that is the root cause of several downstream problems. But that is a ledger design problem, not an annotation problem.

Recommendation

Keep this exactly. It is the most defensible design decision in the project. If anything, the successor product should treat this as the fixed point and redesign everything else around preserving its simplicity.

@file placeholder in the annotation

The abstract placeholder that must be resolved to an actual path at ledger time
Cut or make invisible ▾

Necessary?

No. The file path is already known at annotation time — it is the file the developer is editing. @file solves a problem (not knowing the path) that does not exist for a human writer, and creates a problem (deferred resolution) that did not exist before.

Friction?

High, disproportionately so. It introduces a two-state annotation (unresolved / resolved), requires a ledger pass to complete, and creates a silent failure mode when @file is appended to the ledger verbatim. No other tracing convention uses this pattern.

Value vs. steps?

Negative. The stated benefit is portability — the annotation works even if the file moves. But the resolved ledger entry then becomes stale on a move anyway. The placeholder gives the illusion of resilience without providing it.

@file is the most problematic design decision in Reqtrace v1. The intent was to make annotations location-agnostic — the developer writes the marker without needing to type the full path, and the path is resolved at ledger time via grep. But this creates three costs: (1) every annotation exists in an invalid intermediate state until the ledger pass runs; (2) if the ledger pass never runs or the developer forgets to resolve @file, the ledger contains a broken entry that looks valid; (3) the validator must explicitly reject the literal string @file to catch this failure — a negative check that requires documentation to even explain.

The apparent benefit — not having to type the path — is extremely thin. A developer who is editing examples/refresh-token/src/validation.js knows the path. An AI agent certainly knows the path. The argument that paths change on file rename is also invalid, because the resolved ledger entry changes on rename too. @file defers a known value through an indirection layer that adds a new failure mode for zero resilience gain.

Recommendation

Replace @reqtrace REQUIREMENT/ORDINAL/@file with @reqtrace REQUIREMENT/ORDINAL and make the path implicit from context — it is always the file the annotation lives in. The validator resolves the path from the grep match location. The ledger entry is then generated directly from grep output with no manual resolution step. This eliminates the two-state annotation, the manual ledger pass, and the @file verbatim failure mode in one change.

Ordinal (per-requirement sequence number)

The numeric suffix that identifies one implementation occurrence under a requirement
Rework ▾

Necessary?

Partially. A requirement often has multiple implementation points (model layer, validation, test). You need a way to distinguish them. But a manual sequence number is not the right mechanism — it creates coordination overhead without being stable.

Friction?

High. Developers must know what ordinals already exist before assigning a new one. Two developers working in parallel on the same requirement will collide. The ordinal carries identity in the ledger but provides no protection against duplication.

Value vs. steps?

Low. The ordinal's only job is distinguishing multiple implementations of the same requirement. This could be done by the file path alone — two annotations in different files are already distinguishable without a numeric suffix.

The ordinal is the second most problematic design decision in Reqtrace v1. It is trying to solve a real problem (multiple implementation points per requirement) with a mechanism that requires global coordination between developers. In a solo workflow or an AI-agent workflow with a single actor, ordinals work. In any parallel development context, they are a collision surface.

The deeper issue is that the ordinal encodes sequence, but sequence is not the semantically meaningful dimension. What you actually want to express is: "this is one of multiple places where requirement X is satisfied." The path already encodes which file. The line number (or function name) already encodes which location within the file. The ordinal adds a third identifier that duplicates what context already provides, while creating a new collision risk.

Recommendation

Replace the manual ordinal with an auto-assigned short hash derived from the file path and line number, generated by the validator at ledger-write time. Developers write only @reqtrace REQUIREMENT. The validator, when it generates the ledger entry, appends a stable ID derived from context: AUTH-SESSION-ROTATION/a3f2/examples/refresh-token/src/validation.js:42. This is stable across renames only if the annotation moves with the code, which is the correct behavior, and eliminates all manual ordinal coordination.

The trace ledger (docs/requirements.md)

The canonical human-readable record of validated traces, appended manually after the ledger pass
Expand scope ▾

Necessary?

Yes — the concept is sound. Without a ledger, you have annotations in code but no authoritative record of which annotations have been validated. The ledger is what makes a trace "complete" vs. merely "placed."

Friction?

Very high. The ledger pass is entirely manual: grep, read, assess, append. There is no schema, no structure, no tool to read the ledger back. The result is a free-form document that is maintained by discipline alone and provides no machine-readable signal.

Value vs. steps?

The concept has high value; the implementation has low value. A ledger that only a human can read and only a human can write cannot be validated, queried, or reported on. It is a log, not a record.

The ledger design reveals the core tension in Reqtrace v1: it wants to be grep-native (no tools required) but the ledger pass is the exact point where grep-native breaks down. You cannot grep the ledger to find incomplete traces, detect stale entries, or produce a coverage report — because the ledger has no schema. The ledger is also scope-limited to requirements, but the same tracing pattern is clearly useful for decisions, architecture choices, compliance rules, and test specs. Keeping the ledger requirement-only while those other artifact types exist in the same project is an artificial constraint.

The more fundamental problem: the ledger and the source annotations are two independent records of the same fact. Keeping them synchronized requires human discipline, which is exactly what Reqtrace claims to replace with a convention. This is an internal contradiction in the v1 design.

Recommendation

Auto-generate the ledger from the annotations via the validator, rather than requiring manual append. The developer validates intent; the tool records the result. This eliminates the manual ledger pass. Separately: expand ledger scope beyond requirements to any named handle — decisions, ADRs, specs, policies — using the same annotation syntax. The ledger becomes a project-wide evidence index, not a requirements-only document. Restrict "requirements" to the handle naming convention, not the scope of what is traceable.

Two-stage loop (implement → ledger pass)

The mandatory sequence: implement and annotate first, then grep-validate and record
Rework ▾

Necessary?

The separation of annotation and validation is the right instinct — you do not want validation to slow down implementation. But "mandatory" with no enforcement is not a stage; it is a guideline that will be skipped under pressure.

Friction?

High. The ledger pass is a context switch: stop coding, grep, assess each hit, open the ledger file, append. For a human developer this is a meaningful interruption. For an AI agent this is a separate invocation with no automated trigger.

Value vs. steps?

The validation step has high value. The manual-pass mechanism has low value. The value is separable from the mechanism.

The two-stage loop is structurally correct — annotation first, validation second — but the implementation relies entirely on developer discipline. There is no enforcement boundary between stage one and stage two. A commit with stage-one annotations but no stage-two ledger entries is structurally indistinguishable from a commit where the ledger pass was completed. The "mandatory" label in the README is aspirational, not enforced.

The loop also collapses two distinct concepts: validation (does this annotation correctly trace to the requirement?) and recording (is the trace in the ledger?). In v1, a developer validates with their own judgment and records by appending text. In a successor product, validation should be a separate, explicit step with a defined output — not just "read the requirement and decide."

Recommendation

Formalize the two stages as: (1) annotate in code — unchanged; (2) run the validator, which generates or updates the ledger automatically from validated annotations. The developer's validation judgment is exercised during code review, not as a manual pre-append step. The CI check then enforces that every annotation in the codebase has a corresponding ledger entry — no annotation can be in an unresolved state in a merged commit.

Grep-native design principle

The commitment that traces are findable with standard tools — no graph server, no database
Keep ▾

Necessary?

Yes, as a design constraint. The moment you require a running service to query traces, you've added an infrastructure dependency and broken CI portability. Grep-first means the system degrades gracefully to zero tooling.

Friction?

None. Grep-native is a design principle, not a user step. It constrains what the tool can add without adding user cost.

Value vs. steps?

Very high. This is the core competitive differentiator against heavyweight tracing tools (Jama, Polarion, IBM Doors) and database-backed alternatives. It makes the system universally adoptable.

This is the second strongest design decision in Reqtrace v1. The grep-native principle is what makes the tool usable in air-gapped environments, legacy codebases, and AI agent contexts where spinning up a service is not practical. It is also what makes the tool auditable — anyone can verify the traces by running grep without trusting a tool's output.

The risk is that "grep-native" gets used as a reason to avoid adding any tooling at all, which is the trap v1 fell into with the manual ledger pass. Grep-native should mean "grep is always sufficient to find traces" — it should not mean "the only automation is grep." The validator and a CI hook can coexist with the grep-native principle, because grep still works independently of them.

Recommendation

Keep the grep-native principle as a hard constraint: every trace must be findable with a plain grep command, with no tool installed. But decouple it from the anti-tooling stance. The successor product should be grep-native by design and tool-augmented by choice. Automation adds speed and enforcement; grep remains the ground truth.

Python validator (optional, validate_reqtrace.py)

The dependency-free script that expands @file and checks ledger entries
Rework ▾

Necessary?

The validator's existence is necessary. Its optional status is not. Calling it "optional" while describing the ledger pass as "mandatory" is an internal contradiction that signals the feature was added late and not fully integrated into the design.

Friction?

Low to run (zero dependencies, plain Python). High to adopt — because there are no CI templates, no pre-commit hooks, and no documentation on where in the workflow to run it. "Optional" is the killer: teams skip optional steps under pressure.

Value vs. steps?

High potential, low realized value. The validator could be the enforcement backbone of the entire convention, but in v1 it is a demo script positioned below the convention itself in importance.

The validator's current scope is narrow: expand @file, check ledger entries match. It does not detect stale paths, does not check ordinal uniqueness, does not generate the ledger, and does not know anything about the requirement beyond its handle string. This makes it a consistency checker for the ledger — useful, but not the central tool it should be.

The "optional" framing also sends a signal to AI agents: if AGENTS.md says the ledger pass is mandatory but the validator is optional, an agent will run the ledger pass manually and skip the validator. This is the opposite of what you want — the validator should run automatically and the manual ledger pass should be deprecated in favor of it.

Recommendation

Promote the validator to the primary delivery mechanism. Make it non-optional by providing a pre-commit hook and a GitHub Actions template in the repo. Expand its responsibilities: detect stale paths, detect unresolved @file, detect ordinal collisions, and — in the successor product — auto-generate ledger entries rather than just checking them. Rename it to something that reflects its central role, not a demo script.

AGENTS.md agent guidance

The 16-line instruction file telling AI agents what Reqtrace is and how to use it
Expand scope ▾

Necessary?

Yes. Without explicit agent guidance, AI coding agents will either ignore the convention or invent variations of it. The "Do not invent new canonical requirements in code" rule is the most important constraint in the entire repo — it prevents a very common agent failure mode.

Friction?

None for agents. The file is short, unambiguous, and covers the key prohibitions. It is well-written for its size.

Value vs. steps?

Very high ratio. A 16-line file that prevents agents from hallucinating requirement handles or adding wiki-link syntax is an extremely high-leverage constraint for minimal maintenance cost.

The AGENTS.md is one of the two strongest parts of Reqtrace v1 (alongside the annotation itself). The prohibitions are specific and correct: no claims, no parent fields, no wiki links, no JSON refs, no custom hook names. These are exactly the things a capable AI agent might add to "improve" the convention, and each one would break grep-native compatibility.

The gap is that AGENTS.md covers agent behavior during annotation but says nothing about what agents should do when the ledger needs updating, when a trace is found to be stale, or when running the validator. The ledger pass instructions ("append validated expanded traces") are vague enough that an agent could comply with the letter while violating the intent — appending unresolved @file placeholders, for example.

Recommendation

Expand AGENTS.md to cover three explicit agent scenarios: (1) annotating new code — covered well in v1; (2) running the validator and interpreting its output — currently missing; (3) handling stale or invalid traces found during a ledger audit — currently missing. Each scenario should be a named section with step-by-step instructions. The file can grow to 50 lines without losing its value; what matters is specificity, not brevity.

Requirement-only handle scope

The constraint that @reqtrace handles must reference upstream requirements, not other artifact types
Expand scope ▾

Necessary?

No. The tracing pattern is equally useful for architecture decisions (ADRs), security controls, compliance rules, test specs, and design constraints. Restricting it to requirements is an artificial boundary that forces teams to choose between tracing requirements and tracing other important artifacts.

Friction?

The restriction creates friction in real projects where the implementation evidence for a security control, for example, is more important to trace than the evidence for a functional requirement. Teams either extend the convention informally or don't adopt it for non-requirement tracing needs.

Value vs. steps?

The restriction reduces value without reducing steps. The annotation syntax works identically for any named handle. The only cost of allowing broader handles is updating AGENTS.md and removing the word "requirement" from the nomenclature.

The "requirements only" constraint made sense when Reqtrace was explicitly positioned as a post-requirements-exists tool — it starts after a requirement handle already exists. But this framing conflates the tool's starting condition (a handle exists) with the type of artifact the handle can reference. There is no technical reason why @reqtrace ADR-0012/001 or @reqtrace SEC-CONTROL-7/001 would not work identically to a requirement handle. The grepping works. The ledger works. The validator can work. Only the name "Reqtrace" and the documentation restrict the scope.

Recommendation

In the successor product, remove the requirement-only constraint at the design level. Allow any named handle from any upstream artifact type. The convention becomes: "if a project document makes a claim that implementation should satisfy, that claim can be traced." This broadens the addressable use case from requirements management (a niche) to general implementation evidence tracing (a universal need). Consider renaming away from "Reqtrace" to reflect the expanded scope.

Roadmap features — assessment

Skill automation for AI agents

A coding agent skill that automates the annotation and/or ledger pass during AI-assisted implementation
Rework the approach ▾

Necessary?

Yes — but only if the underlying annotation and ledger design is simplified first. Automating a flawed convention produces a flawed automation. If @file resolution and the manual ledger pass survive into the successor, the skill will just automate the most error-prone parts of a fragile workflow.

Friction?

Depends entirely on what the skill does. A skill that writes annotations is low-friction. A skill that runs the ledger pass — greps, reads, validates, appends — is high-friction for the agent and high-latency for the developer. The two should be separate skills.

Value vs. steps?

Very high if scope is narrowed to annotation placement only. Medium if it includes ledger management. The skill should do what humans are worst at: remembering to annotate every relevant location. Not what they are best at: deciding whether a trace is valid.

Skill automation for Reqtrace is the right roadmap direction but carries a sequencing risk: if the annotation simplification (removing @file, replacing manual ordinals) is not shipped before or alongside the skill, the skill will encode the v1 complexities into an automated workflow that is then hard to change. Agents are good at placing annotations near code they understand; they are poor at knowing whether the trace is semantically valid without reading the requirement.

The most valuable skill is not the ledger-pass automation — it is the annotation-placement suggestion: "you just implemented AUTH-SESSION-ROTATION behavior at line 47; do you want to add a trace annotation?" This keeps the human in the validation loop while removing the cognitive overhead of remembering to annotate.

Recommendation

Build two separate skills: (1) annotation placement — the agent suggests annotations based on requirement context when implementing code, and places them on human confirmation; (2) ledger audit — the agent runs the validator, reports unresolved annotations, stale paths, and ordinal collisions, and auto-generates ledger entries for confirmed valid traces. Keep semantic validation (is this trace actually correct?) as a human step. Don't automate judgment.

CI/pre-commit enforcement (roadmap)

Automated enforcement that all annotations are resolved and all ledger entries are valid on commit or push
Ship earlier than planned ▾

Necessary?

Yes, and more urgently than the skill automation. Without CI enforcement, the "mandatory ledger pass" claim has no teeth. Every team will accumulate unresolved annotations. The validator exists but the enforcement hook does not.

Friction?

Low once set up. A pre-commit hook that runs the validator in under two seconds adds negligible friction. The setup cost is a one-time investment in a GitHub Actions template and a .pre-commit-hooks.yaml entry.

Value vs. steps?

Very high. CI enforcement is what turns a convention into a guarantee. Without it, Reqtrace is "the tracing convention that we try to follow." With it, Reqtrace is "the tracing convention that we cannot violate."

CI enforcement should be considered a v1 gap, not a roadmap feature. Its absence is the primary reason the ledger pass is frequently skipped in practice. The validator already does the checks — it just needs to be wired into CI. This is two files: a .github/workflows/reqtrace.yml that runs python scripts/validate_reqtrace.py --strict and a .pre-commit-hooks.yaml entry. Both could be added to the repo today.

Recommendation

Ship the CI template and pre-commit hook immediately, before the skill automation. The sequence should be: fix @file → add auto-ordinals → auto-generate ledger → CI enforcement → skill automation. Enforcement without the prior simplifications will enforce a fragile workflow, which is worse than no enforcement because it breaks CI on legitimate work.

Coverage reporting (roadmap)

A report of which requirement handles have zero, partial, or full trace coverage in the codebase
High value — prioritize ▾

Necessary?

Yes, for any team using Reqtrace at scale. "Which requirements have no implementation evidence?" is the question that makes tracing valuable in the first place. Without a coverage answer, the ledger is just a log.

Friction?

None for users if the report is auto-generated. The complexity is in the implementation: the reporter needs to read the canonical requirement list and cross-reference against the ledger. This requires the ledger to have a schema — which is itself a blocker if the ledger remains free-form text.

Value vs. steps?

Extremely high. Coverage reporting is the feature that turns Reqtrace from a tracing convention into a project management signal. "3 of 12 requirements have no implementation evidence" is actionable information that the annotation syntax alone cannot provide.

Coverage reporting is the killer feature that v1 cannot deliver because the ledger has no schema. To report coverage, you need: (1) a canonical list of requirement handles, (2) a machine-readable ledger, and (3) a diff between them. V1 has (1) implicitly (the requirement handles in the code), but not (2) or (3). The ledger is free-form text that cannot be reliably parsed. This is the strongest argument for defining a ledger schema as a prerequisite to everything else on the roadmap.

Recommendation

Make coverage reporting the north star feature that backward-constrains the ledger design. The question to answer is: "what ledger format makes coverage reporting easiest to implement and most useful to read?" A structured ledger (YAML or JSON sidecar, or a fenced Markdown table) answers this. Design the ledger schema for the report you want, not for the annotation you have. Then everything else — the validator, the skill, the CI check — serves the coverage report.
